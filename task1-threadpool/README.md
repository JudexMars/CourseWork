# Задание 1: Кастомный Thread Pool — отчёт

## 1. Архитектура решения

Реализован пул потоков `CustomThreadPool`, реализующий интерфейс `CustomExecutor`. Основные компоненты:

| Компонент | Описание |
|-----------|----------|
| `CustomExecutor` | Контракт: `execute`, `submit`, `shutdown`, `shutdownNow` |
| `CustomThreadPool` | Управление воркерами, очередями и политикой отказа |
| `Worker` | Рабочий поток, привязанный к одной очереди |
| `NamedThreadFactory` | Создание потоков с именами `MyPool-worker-N` |
| `PoolConfig` | Параметры пула (record) |
| `RejectionPolicy` | Политики отказа: `ABORT`, `CALLER_RUNS`, `DISCARD` |

### Round Robin — распределение задач

У каждого воркера своя `ArrayBlockingQueue` фиксированной ёмкости `queueSize`. При вызове `execute()` пул выбирает очередь по формуле:

```
queueIndex = roundRobinCounter++ % workerCount
```

Если выбранная очередь заполнена, пул пробует следующую (циклический обход). Это обеспечивает равномерное распределение нагрузки между воркерами без центральной очереди.

### minSpareThreads

Если число свободных (не занятых задачей) воркеров падает ниже `minSpareThreads`, пул создаёт новый воркер — даже при низкой общей нагрузке. Это гарантирует наличие «резервных» потоков для быстрого отклика на всплески запросов.

### Idle timeout

Воркер ожидает задачу через `queue.poll(keepAliveTime, timeUnit)`. При таймауте, если `activeWorkers > corePoolSize`, поток завершается с логом `[Worker] ... idle timeout, stopping.`

### Логирование

Все ключевые события фиксируются через `java.util.logging`:

- `[ThreadFactory]` — создание потока
- `[Pool]` — постановка задачи в очередь, shutdown
- `[Worker]` — выполнение задачи, idle timeout, завершение
- `[Rejected]` — отказ при перегрузке

## 2. Политика отказа: ABORT

Выбрана политика **ABORT** — при переполнении всех очередей и занятости всех воркеров выбрасывается `RejectedExecutionException`.

**Обоснование:**
- Предсказуемое поведение для высоконагруженного сервера
- Клиент сразу узнаёт о перегрузке и может применить retry/backpressure
- Не блокирует вызывающий поток (в отличие от `CallerRunsPolicy`)

**Недостатки:**
- Задача теряется без дополнительной обработки на стороне клиента
- Требует внешней логики повторных попыток

**Альтернативы:**
- `CALLER_RUNS` — выполняет задачу в потоке вызывающего; снижает throughput submitter'а, но не теряет задачу
- `DISCARD` — молча отбрасывает; подходит только для некритичных задач

Демонстрация: при `core=2, max=2, queueSize=2` из 25 задач принято 6, отклонено 19.

## 3. Сравнение производительности

Бенчмарк в `Main.runBenchmarkComparison()`:

- **500 задач**, каждая спит **10 мс** (имитация IO-bound работы)
- Параметры: `core=4, max=8, queueSize=150, minSpare=0, keepAlive=60s`
- Метрики: throughput (tasks/sec) до завершения всех `Future`, средняя latency (ms) на задачу
- Среда: macOS, Java 21, полный прогон `./gradlew :task1-threadpool:run` (13.06.2026, 21:53)

### Результаты

| Пул | Throughput | Avg latency |
|-----|------------|-------------|
| CustomThreadPool | **344.23** tasks/sec | **11.02** ms |
| ThreadPoolExecutor | **355.79** tasks/sec | **10.72** ms |

### Анализ

`ThreadPoolExecutor` с единой очередью (`ArrayBlockingQueue(1200)`) оказался быстрее примерно на **3.4%** по throughput (355.79 vs 344.23 tasks/sec). Разница в latency минимальна (~0.3 ms).

Причины небольшого отставания `CustomThreadPool`:
- Round Robin и обход per-worker очередей под `synchronized(workersLock)` при массовой подаче задач
- Создание воркеров «на лету» при заполнении очередей

Преимущество `CustomThreadPool` — изоляция нагрузки: переполнение одной очереди не блокирует весь пул, задачи перенаправляются на другие воркеры.

Запуск только бенчмарка:

```bash
./gradlew :task1-threadpool:run --args="benchmark"
```

Полная демонстрация:

```bash
./gradlew :task1-threadpool:run
```

## 4. Исследование параметров

Эксперименты (`Main.runParameterResearch()`): **200 задач × 10 ms**, `queueSize=50`, варьируются `corePoolSize` и `maxPoolSize`.

### Результаты

| core | max | queueSize | Throughput (tasks/sec) |
|------|-----|-----------|------------------------|
| 2 | 4 | 50 | **332.95** |
| 4 | 8 | 50 | **346.14** |
| 8 | 16 | 50 | **706.12** |

### Выводы

1. Для IO-bound задач (sleep 10 ms) throughput растёт с числом потоков — при `8/16` результат почти вдвое выше, чем при `2/4`, так как потоки большую часть времени ждут и не конкурируют за CPU.
2. Переход `2→4` потоков даёт минимальный прирост (~1.4%), узкое место — не число потоков, а накладные расходы на постановку задач.
3. Скачок до **706 tasks/sec** при `core=8` показывает, что для блокирующих задач увеличение пула оправдано до тех пор, пока задачи преимущественно IO-bound.
4. Для **CPU-bound** задач оптимум ближе к `corePoolSize ≈ число ядер`; дальнейшее увеличение потоков не даст прироста.
5. **minSpareThreads > 0** полезен для latency-sensitive сервисов, но расходует память на простаивающие потоки.
